# 中文写作规范标准（完整版）

> 来源：汉洋《写作入门》、汉洋《给人文工作者的 AI 使用指南》、Geedea《论遣词造句》、[中文文案排版指北](https://github.com/sparanoid/chinese-copywriting-guidelines)

---

## 一、标点符号

### 1.1 中文标点

| 规则 | ✅ 正确 | ❌ 错误 |
|------|---------|---------|
| 一律使用全角标点 | `，。！？；：` | `,. ! ? ; :` |
| 使用直角引号，「」为一级，『』为二级 | `他说「你好」` | `他说"你好"` |
| 破折号用两个 em dash | `——` | `--` 或 `—` |
| 省略号用两个英文省略号 | `……` | `。。。` 或 `......`（六个点） |
| 用斜杠表示"或者"时用 `/` | `他/她` | `他\她` |

**波浪线 `～` 使用提示**：偶尔使用可增加语气变化，但不可滥用——"调味料放多了会咸"。

### 1.2 英文标点

| 规则 | ✅ 正确 | ❌ 错误 |
|------|---------|---------|
| 一律使用半角 | `Hello, World.` | `Hello，World。` |
| 逗号/分号/冒号/句号/感叹号/问号后留一个空格 | `Hello, World` | `Hello,World` |
| 省略号是一个符号 `…` | `Wait…` | `Wait...`（三个点） |
| 破折号用 em dash `—`（Mac: Shift+Option+-） | `She — the boss — left.` | `She -- the boss -- left.` |
| 表示范围用 en dash `–`（Mac: Option+-） | `1980–1992` | `1980-1992` |
| 括号外面留空格，里面不留 | `James (aka Aphex Twin) is` | `James( aka Aphex Twin )is` |
| 大小写必须正确 | `GitHub`, `iPhone` | `Github`, `Iphone` |

### 1.3 标点通用规则

| 规则 | ✅ 正确 | ❌ 错误 |
|------|---------|---------|
| 不重复使用标点符号 | `战胜了巴西队！` | `战胜了巴西队！！！` |
| 全角标点与其他字符之间不加空格 | `刚买了一部 iPhone，好开心！` | `刚买了一部 iPhone ，好开心！` |
| 数字使用半角字符 | `这个蛋糕只卖 1000 元` | `这个蛋糕只卖 １０００ 元` |
| 完整英文句子内使用半角标点 | `「Stay hungry, stay foolish.」` | `「Stay hungry，stay foolish。」` |

## 二、中英文混排

### 2.1 空格规则

| 规则 | ✅ 正确 | ❌ 错误 |
|------|---------|---------|
| 中文与英文之间加空格 | `使用 GitHub 管理代码` | `使用GitHub管理代码` |
| 中文与数字之间加空格 | `现在是 2016 年` | `现在是2016年` |
| 中文与链接之间加空格 | `请访问 https://example.com 了解` | `请访问https://example.com了解` |
| 数字与单位之间加空格 | `宽带有 10 Gbps，SSD 有 20 TB` | `宽带有 10Gbps，SSD 有 20TB` |
| 度数/百分比与数字之间**不**加空格 | `角度为 90°` / `提升 15%` | `角度为 90 °` / `提升 15 %` |

### 2.2 专有名词大小写

务必保持专有名词的正确大小写，常见易错示例：

| ✅ 正确 | ❌ 错误 |
|---------|---------|
| `GitHub` | `Github` / `github` / `GITHUB` |
| `iPhone` | `Iphone` / `iphone` / `IPHONE` |
| `JavaScript` | `Javascript` / `javascript` |
| `macOS` | `MacOS` / `MACOS` |
| `Wi-Fi` | `wifi` / `WIFI` / `Wifi` |
| `iOS` | `IOS` / `ios` |

### 2.3 不要使用不地道的缩写

| ✅ 正确 | ❌ 错误 |
|---------|---------|
| `TypeScript` | `Ts` |
| `HTML5` | `h5` |
| `React` / `Next.js` | `RJS` / `nextjs` |
| `前端开发者` | `FED` |

### 2.4 产品名按官方格式书写

产品名应严格按照官方定义的格式书写，即使可能违反空格等通用规则：

- `豆瓣FM`（官方无空格，按官方格式）
- `LeanCloud`（官方格式）

### 2.5 语言混杂原则

- **能用中文表达清楚的，优先使用中文**
- 专有名词（产品名、品牌名、技术术语）无广泛接受的中文译名时，可保留原文
- **不要为了显示"高级感"而中夹英**
- 如需使用英文术语，首次出现时给出中文解释

| ✅ 正确 | ❌ 错误 |
|---------|---------|
| `使用 Docker 容器化部署` | `把这个 project deploy 到 server 上`（可以全中文说清楚的不必夹英） |
| `由拉丁语演变而来的语言` | `拉丁语系`（不存在的术语） |

## 三、数字用法

| 规则 | ✅ 正确 | ❌ 错误 |
|------|---------|---------|
| 一位数使用汉字 | `山药至少有三种常见的做法` | `山药至少有3种常见的做法` |
| 两位以上使用阿拉伯数字 | `一年有 365 天` | `一年有三百六十五天` |
| 阿拉伯数字前后加空格 | `共 12 人参加` | `共12人参加` |

## 四、用词规范

### 4.1 禁用词汇

以下词汇在除了直接引语以外的任何情况下禁止使用：

| 禁用词 | 说明 |
|--------|------|
| 屌丝 | 粗俗且已过时 |
| 白富美 / 高富帅 | 刻板标签 |
| 高大上 | 空洞无实义 |
| 富二代 / 红二代 / 军二代 | 标签化用语 |
| 安利（动词用法） | 网络用语，意义模糊 |
| 撕逼 | 粗俗用语 |
| 尼玛（语气助词） | 粗俗变体 |
| 也是醉了 | 陈词滥调，应具体描述感受 |
| 有木有 | 刻意卖萌，用「有没有」 |
| 美女 / 帅哥（称呼用） | 不正式 |
| 给力 | 已过时的网络用语 |

**常识性原则**：如果把某个词用汉字完整写出来令你不适，那么也不要用柔化过的变体——**不要用这个词**。

> ❌ 「关我 P 事」——写成缩写并不会让你更不粗俗，请直接不用这个表达。

### 4.2 慎用词汇

以下词汇容易空洞或被滥用，使用时需确认是否真的恰当：

| 慎用词 | 注意事项 |
|--------|----------|
| 情怀 | 确认不是在贩卖空洞的感情 |
| 物欲横流 | 老套，考虑更具体的表达 |
| 众所周知 | 真的众所周知吗？ |
| 简约 | 太泛，不如具体描述何种简约 |
| 唯美 | 空洞修饰语 |
| 精品 | 如果人人都说自己是精品就没有精品了 |
| 行为艺术 | 这是一种正式的艺术类别，不能描述一切你看不懂的东西 |
| Low | 如果觉得某物 low，请具体描述：为什么 low？是哪种 low？ |
| 人工智能 | 除特定宣传场合外，应使用具体技术名称（如大语言模型、文生图模型等） |

### 4.3 避免陈词滥调

- 当成语或惯用语第一时间跳入脑中时，停下来思考是否过于泛滥
- 被滥用的表达即便准确，也会失去效力——文字和食物一样，新鲜是第一要素

| ❌ 陈词滥调 | 💡 建议 |
|------------|---------|
| 「一千个人眼里有一千个哈姆莱特」 | 直接阐述你要表达的观点 |
| 「并没有什么卵用」 | 「并没有什么用」即可，不损失信息 |
| 「万字长文」「深度好文」 | 长和好没有关联，依靠文字本身吸引读者 |

### 4.4 避免网络热词

- 不要因为某个词汇或句式在当下流行就忙不迭地使用
- 不要把互联网上的黑话带到正式写作中
- 使用最基本的词句表达想法，遇到需要解释的概念不吝啬笔墨

### 4.5 专有名词须查证

- 遇到专有名词时，必须查阅可靠资料了解准确含义，**不能从字面意思推断**
- 不要因为经常听到某个词就假定它是正确的

| ❌ 错误用法 | ✅ 正确用法 | 说明 |
|-----------|-----------|------|
| `XXX 是个伪命题` | `XXX 是个不存在的概念` / `XXX 这个说法站不住脚` | 「伪命题」指假命题，不是「不存在的概念」 |
| `拉丁语系` | `罗曼语族` / `由拉丁语演变而来的语言` | 「拉丁语系」在语言学分类中不存在 |

## 五、避免欧化中文

### 5.1 减少不必要的量词和代词

中文中不需要像英文那样频繁使用冠词/代词。如非强调需要，应删去多余的「一个」「一种」「这个」「那些」等。

| ❌ 欧化表达 | ✅ 自然中文 |
|-----------|-----------|
| `这是一种非常不负责任的行为` | `这非常不负责任` |
| `我们需要一个更好的解决方案` | `我们需要更好的解决方案` |
| `这个项目的这些问题需要解决` | `项目中的问题需要解决` |

**进阶建议**：如果非要用指示词，可以换用不同表达避免重复——「上述」「此类」「前者」等。

### 5.2 避免抽象名词泛滥

中文天然偏好动词表达，避免将本可以用动词表述的内容名词化。

| ❌ 名词化 | ✅ 动词化 |
|---------|---------|
| `对该问题的解决` | `解决该问题` |
| `进行了充分的讨论` | `充分讨论了` |
| `做出了重要的决定` | `做了重要决定` |

### 5.3 避免不必要的音译

中文有能力直接表达的概念，不应使用英语音译来追求"高级感"。

| ❌ 不必要的音译 | ✅ 正确中文 |
|---------------|-----------|
| `芝士` / `起司` | `奶酪` |

但以下情况例外：
- 已被广泛接受且无歧义的音译（如"巧克力""咖啡"）
- 品牌名和产品名保留原文

## 六、文风规范

### 6.1 禁止的文风类型

| 文风类型 | 示例 | 问题 |
|----------|------|------|
| 新闻联播体 | `事情真的像用户想像的那样吗？带着这个问题，记者走访了……` | 不适合科技公司或个人写作 |
| 淘宝体 | `亲，这个 Bug 修改了，请给我们点个赞哦` | 不专业 |
| 自嗨型 | 只关注自己想说的，不考虑读者是否在乎 | 读者没有义务读你的文字 |

### 6.2 应遵循的文风原则

1. **说人话**：明白、清晰、准确。「好吃不贵」就是最好的人话。
2. **别写自己不信的东西**：人在说自己不信的东西时最容易拽大词。
3. **不要耍小聪明**：应用写作的目的是沟通，不是显示文笔、专业知识或"脑洞"。
4. **清晰比聪明更好**（Clear is better than clever）——编程领域的谚语，同样适用于自然语言。
5. **不要害怕崇高**：言不由衷的崇高是虚伪，害怕崇高是犬儒。
6. **长文和好文没关系**：长和好从来没有关联关系，依靠文字本身吸引读者。

### 6.3 翻译/双语检查

- 翻译完毕后务必二次检查英文是否反映中文含义
- 确保英文文档经过英语母语者审阅
- 不同语言的微妙之美是机器翻译难以实现的

## 七、诚恳写作原则

### 7.1 不要高估共性

- 不要以为用自己习惯的表达，所有人都能听懂
- 撰写长文时，把前提交代清楚，把讨论基础打好
- 使用最基本的词句表达想法，遇到概念不吝啬笔墨解释

### 7.2 不要高估理解力

- 遇到专有名词，查阅可靠资料而非从字面推断
- 不熟悉的专有名词，宁可不用也别乱用
- 如果不是某领域专家，不要乱用该领域的术语；用更基础的语言描述即可

### 7.3 为读者而写

- 假定没人在乎你写的东西——给读者至少一个阅读理由
- 竞争对手不是另一篇文章，而是短视频、爽文和线下生活
- 写完之后大声朗读，你会在朗读中发现绝大部分问题
